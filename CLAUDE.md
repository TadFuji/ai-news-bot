## ディレクトリ専用ルール R1：徹底的なバックアップ

**何かを変更・上書き・削除する前に、必ずバックアップを取り、その存在を目視確認してから進めること。**
藤川さんはコードを読まないため、「いつでも元に戻せる状態」を物理的に担保することが唯一の安全網です。
バックアップ未確認のまま破壊的な操作には進みません。

### 必ずバックアップする対象（かけがえのない本番データ）

1. `ai_news.db`（SQLite データベース＝蓄積された記事データ）
2. `docs/` 配下の JSON（公開中の朝刊・コラムデータ。130 日分以上が蓄積。再生成できない過去データ）
3. `.env`（秘密情報。中身は表示・複製の際に必ずマスクし、外部やコミットに出さない）
4. これから編集・上書きする全てのソースファイル

### 手順

1. 着手前に作業ツリーがクリーンかを確認し、現在のコミット ID（変更の目印）を控える。git 自体が第一のロールバック層。
2. 編集・上書き・削除の前に、対象を `.backups/<YYYY-MM-DD_HHMM>/`（タイムスタンプ付きフォルダ）へコピーする。
3. コピー後、バックアップが実在することをファイル一覧で確認してから本作業に入る。
4. 作業後に「どのバックアップを・どこへ戻せば元に戻るか」を一文で残す（ロールバック手順）。

### やってはいけないこと

- バックアップ未確認のままの上書き・削除・DB 変更・force push（強制的な履歴上書き）。
- `git add .` / `git add -A`。クラウド同期下で重複ファイル（`xxx (1).ext`）が混入するため、対象を **個別指定** する。
- 直接の削除（`rm` / `del` / `Remove-Item` 等）。削除はグローバル `CLAUDE.md` §4 の `recycle.ps1` でごみ箱へ送る。
- `.backups/` を git にコミットすること。秘密情報を含み得るため、`.gitignore` で除外済み。
